# 개발 외주 자동 견적 사이트 — 개발 명세서

> Claude Code 작업용 문서. 이 문서 하나로 기획·UX·가격 로직·데이터·빌드 순서를 모두 파악할 수 있도록 작성됨.
> 작성 기준일: 2026-05-29

---

## 1. 한 줄 요약

비개발자 클라이언트가 **"집 짓기"에 빗댄 직관적인 화면**에서 선택만으로 **실시간 자동 견적**을 받는, 1인(또는 소규모 팀) 외주 수주용 단독 사이트.

---

## 2. 프로젝트 성격 (확정 사항)

| 항목 | 결정 |
|------|------|
| 사이트 형태 | 중개 플랫폼 X / **본인(팀)이 외주를 받는 단독 사이트** |
| 견적 방식 | **실시간 자동 금액 산출** (정보만 수집하는 리드폼 아님) |
| 가격 포지션 | 시장 대비 **저가** (포트폴리오·얼리버드 가격) |
| 개발 방식 | 전 과정 **바이브코딩(Claude Code)** |
| 운영 주체 | 학부생 1인, 외주 무경험 → 캐파(capacity) 제한 고려 필요 |

---

## 3. 포지셔닝 전략 (사이트 카피·정책에 반영)

1. **저가는 "포트폴리오·얼리버드 가격"으로 프레이밍.**
   "정상가 대비 도입가" 식으로 노출 → 레퍼런스 쌓인 뒤 가격을 올려도 스스로를 싼값에 가두지 않음.

2. **바이브코딩은 약점이 아니라 무기.**
   핵심 카피: **"AI로 빠르게 만들고, 보안 전공자가 직접 검수합니다."**
   - 비개발자가 가장 불안해하는 'AI 코드 신뢰' 문제를 정면 해소
   - 싼 가격의 근거(빠름) + 신뢰의 근거(사람 검수)를 한 문장으로 전달
   - `보안 정밀 점검`을 견적 옵션으로 제공 (운영자의 보안 전공 활용)

3. **진짜 리스크는 낮은 가격이 아니라 범위 추가(scope creep).**
   견적 결과에 항상 **"포함 / 별도"** 리스트를 노출해 방어.

4. **캐파 가드레일.**
   규모·기한 입력값이 1인이 감당 불가한 수준이면 자동 견적 대신 **"상담 필요"**로 분기.

## 4. 2-레이어 아키텍처 (중요)

선택 화면과 계산 로직을 **분리**한다.

```
[Layer A · 클라이언트 화면]   ──선택값──▶   [Layer B · 내부 견적 엔진]
 집 짓기 비유 / 금액만 노출                  man-day 기반 실제 계산
 (공수·단가 단어 절대 노출 X)                (운영자/내부 로직)
```

- **Layer A (Client View):** 비유 카드 + 예상 금액 + "왜 이 금액?" 설명 + 포함/별도. 전문용어 없음.
- **Layer B (Engine):** 모듈별 man-day × 인일단가 × 4대 분기 가중치 × 버퍼. 공수가 그대로 일정으로 변환됨.
- 동일한 선택값(state)이 A에서 입력되어 B 엔진으로 계산된다. 클라이언트에게는 B의 내부 수치를 노출하지 않는다.

## 5. 견적 입력 항목 (Configurator Spec)

### 5.1 4대 분기 (가중치로 작동)

| 분기 | 선택지 |
|------|--------|
| 플랫폼 | 웹(반응형) / 모바일 앱 |
| 대상 | 내부용 / 외부용 |
| 사용자 규모 | 소(~1천) / 중(~10만) / 대(10만+) |
| 코드 | 신규 개발 / 기존 코드 리팩토링 |
| 긴급도 | 여유 / 급행 |

### 5.2 기능 모듈 (선택식 — 견적의 핵심 변수)

회원·로그인 / 결제 / 실시간 소통(채팅) / 관리자 페이지 / 알림 보내기 / AI 기능 / 보안 정밀 점검
(상세 effort 값은 §6.3 데이터 정의 참고)

### 5.3 사용자 개인 섹션

- 기한 (deadline) — 긴급도 계수 자동 산출에 사용
- 중간 회의 간격
- **예산 범위** (필수 — 없으면 견적이 떠다님)
- 연락 채널 (Slack / 노션 / 카톡 / 이메일)
- 기존 자료(기획서·디자인·레퍼런스) 유무
- 계약 형태 (고정가 / 시간제)

## 6. 가격 산정 로직 (Layer B 엔진)

### 6.1 산정 근거 = 하이브리드

- **엔진(실시간 계산):** 모듈별 man-day 합산 × 인일단가 × 가중치  ← 실제 금액은 이걸로 계산
- **신뢰 레이어:** 정부 SW사업 대가산정 가이드(기능점수·FP)를 견적서에 "표준 대비 합리적 가격" 근거로 *표시용*만 사용
- **가드레일:** 위시켓 계산기·숨고 범위로 결과가 시장에서 벗어나지 않는지 가끔 검산

> ⚠️ FP 단독 사용 금지(대형 SI 전제라 1인 견적이 2~3배로 튐), 시장평균 단독 사용 금지(본인 거래 데이터 없음).

### 6.2 계산 공식

```
견적 = ( 기본공수 + Σ선택모듈공수 )
        × 플랫폼계수
        × 대상계수
        × 규모계수
        × 코드계수(리팩토링)
        × 긴급도계수
        × 리스크버퍼
        × 인일단가
```

- 클라이언트에는 **원 단위 금액 + ±15% 범위**만 노출. man-day·단가는 숨김.
- 금액은 50,000원 단위로 반올림.
- 부가세 별도 표기.
- 리팩토링 선택 시: 진단 공수 별도 + 버퍼를 1.35로 상향(미지의 변수 반영).

### 6.3 데이터 정의 (구현용 — 그대로 코드로 옮기면 됨)

> 아래 수치는 **저가·바이브코딩** 기준으로 캘리브레이션한 시작값.
> 운영자가 실제 작업 속도(예: 클래스프·시미나 플랫폼 제작 시간)로 1~2개 모듈을 역산해 보정할 것.

```ts
// 내부 엔진 상수
const RATE = 100_000;        // 인일단가(원/일). 운영자 실제 단가로 교체
const BUFFER = 1.15;         // 리스크 버퍼 (리팩토링이면 1.35)
const BASE_EFFORT = 5;       // 설계·기본화면·오픈에 해당하는 기본 공수(일)

// 기능 모듈 공수(effort, 단위: 일)
const FEATURES = {
  login: { label: "회원·로그인",     effort: 2 },
  pay:   { label: "결제",            effort: 4 },
  chat:  { label: "실시간 소통",     effort: 5 },
  admin: { label: "관리자 페이지",   effort: 5 },
  noti:  { label: "알림 보내기",     effort: 1.5 },
  ai:    { label: "AI 기능",         effort: 5 },
  sec:   { label: "보안 정밀 점검",  effort: 3 },
};

// 4대 분기 가중치
const MULTIPLIERS = {
  platform: { web: 1.0, app: 1.5 },
  audience: { internal: 1.0, external: 1.25 },
  scale:    { small: 1.0, mid: 1.2, large: 1.45 },
  base:     { new: 1.0, refactor: 1.35 },   // refactor일 때 BUFFER도 1.35로
  urgency:  { normal: 1.0, rush: 1.25 },
};
```

```ts
// 계산 함수 (참고 구현)
function estimate(state) {
  let effort = BASE_EFFORT;
  for (const k in state.feats) if (state.feats[k]) effort += FEATURES[k].effort;

  const m =
    MULTIPLIERS.platform[state.platform] *
    MULTIPLIERS.audience[state.audience] *
    MULTIPLIERS.scale[state.scale] *
    MULTIPLIERS.base[state.base] *
    MULTIPLIERS.urgency[state.urgency];

  const buffer = state.base === "refactor" ? 1.35 : BUFFER;
  const raw = effort * m * buffer * RATE;
  const price = Math.round(raw / 50000) * 50000;

  return {
    price,
    low:  Math.round(price * 0.85 / 50000) * 50000,
    high: Math.round(price * 1.15 / 50000) * 50000,
    estimatedDays: Math.round(effort * m * buffer),   // 일정 산출용(내부)
  };
}
```

### 6.4 공수 → 일정 자동 연결

- `estimatedDays`(1인 기준) 산출 → 사용자 입력 `기한`과 비교
- 기한이 부족하면 자동으로 긴급도 `rush` 적용, 그래도 불가하면 **"상담 필요"** 분기

## 7. 클라이언트 UX 명세 (Layer A)

전체를 **"집 짓기"** 비유 하나로 통일. 전문용어 노출 금지.

### 7.1 비유 매핑 (선택 카드)

| 분기 | 선택지 | 아이콘(Tabler) | 제목 | 한 줄 비유 |
|------|--------|----------------|------|-----------|
| 플랫폼 | web | `ti-world` | 웹사이트 | 주소만 알면 누구나 와요 |
| 플랫폼 | app | `ti-device-mobile` | 모바일 앱 | 폰에 설치하는 전용 매장 |
| 대상 | internal | `ti-lock` | 우리끼리 (내부용) | 직원 전용 작업실 |
| 대상 | external | `ti-building-store` | 손님 대상 (외부용) | 간판·꾸밈·잠금장치 필요 |
| 규모 | small | `ti-coffee` | 동네 카페 | 소수가 이용 |
| 규모 | mid | `ti-building` | 인기 맛집 | 꽤 많이 몰려요 |
| 규모 | large | `ti-building-skyscraper` | 백화점 | 아주 많이 몰려요 |
| 코드 | new | `ti-square-plus` | 빈 땅에 새로 | 처음부터 깔끔하게 |
| 코드 | refactor | `ti-tool` | 기존 집 리모델링 | 열어보기 전엔 모를 수 있어요 |
| 긴급 | normal | `ti-calendar` | 여유 있게 | 표준 일정 |
| 긴급 | rush | `ti-urgent` | 급해요 (급행) | 집중 투입이 필요해요 |

### 7.4 결과 표시

- **예상 견적:** `약 {price}원` (크게)
- **범위:** `{low} ~ {high}원 · 간단한 상담 후 최종 확정 · 부가세 별도`
- **왜 이 금액인가요?:** §7.3 카피 목록
- **포함 / 별도** 2단 표시 (scope creep 방어 — 항상 노출)

```ts
// 항상 포함
const INCLUDED_BASE = ["설계·기획", "기본 화면 제작", "오픈(배포)", "간단한 사용법 안내"];
// + 선택한 기능 모듈 label 추가

// 항상 별도(안내 후 추가)
const EXCLUDED = ["사진·글 등 콘텐츠 준비", "도메인·서버 월 비용", "오픈 후 새 기능 추가", "지속 운영 대행"];
```

## 8. 차별화 기능 (우선순위순)

1. **보안 감수 옵션** — 시큐어코딩 리뷰·점검을 add-on으로. 운영자 보안 전공 활용. (외부용·결제 선택 시 자동 추천)
2. **AI 요구사항 정리기** — 클라이언트가 "당근마켓 같은 거요" 한 줄 입력 → Claude API가 구조화된 기능 명세 + 해당 옵션 자동 체크. 비전공자 진입장벽 제거.
3. **옵션별 가격 기여도 시각화** — Recharts로 "이 선택이 +N원" 실시간 표시.
4. **견적 PDF·공유링크 발급** — 포함/제외 명세서 포함, FP 근거 표기로 신뢰도.
5. **마일스톤 자동 생성** — 기한 + 회의 간격 역산 일정표.

## 10. 빌드 순서 (Claude Code 단계별)

> 한 번에 다 만들지 말고 단계별로 쪼개서 진행.

- **Phase 1 — 클라이언트 선택 화면 컴포넌트**
  §7 비유 카드 UI. 4대 분기 라디오 카드 + 기능 모듈 체크 카드. 상태(state)만 관리.
- **Phase 2 — 견적 계산 로직**
  §6 엔진 구현. state → estimate(). 결과 카드(금액·범위·왜 이 금액·포함/별도) 연결.
- **Phase 3 — 캐파 가드레일 + 일정 연결**
  §6.4. 기한 대비 공수 비교, 비현실적이면 "상담 필요" 분기.
- **Phase 4 — 결과 산출물**
  견적 PDF / 공유링크 발급.
- **Phase 5 — 견적 저장 백엔드 + 리드 관리**
  Supabase 스키마(견적 레코드, 연락처, 선택 내역 JSON).
- **Phase 6 — 차별화 기능**
  AI 요구사항 정리기, 보안 옵션 강조 카피.

## 11. 참고: 시장 환경 (조사 결과 요약)

- 중개 플랫폼(위시켓·크몽·프리모아·이랜서 등)은 대부분 "연결만 하고 일정·품질·소통은 방치", 유지보수 단계에서 인력 단절 문제가 흔함 → 단독 사이트로 일관된 관리가 차별점.
- 위시켓은 이미 "앱 개발 비용 계산기"를, 숨고는 데이터 기반 가격 범위를, 스마트메이커는 정부 FP 방식을 운영 중 → 자동 견적 자체는 신규성이 낮음. **차별점은 "집 짓기 비유 + 비개발자 언어 설명 + 보안 전공자 검수"의 조합.**
- 핵심 통찰: **견적의 본질은 '금액'이 아니라 '포함 범위'.** 같은 금액도 어디까지 포함하느냐로 완성도가 갈린다 → 포함/별도 명세를 전면에.
