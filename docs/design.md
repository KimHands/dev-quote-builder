## Problem Statement

표면 문제: "비개발자 클라이언트가 집 짓기 비유 화면에서 선택만으로 실시간 자동
견적을 받는 1인 외주 수주용 사이트."

office-hours로 드러난 **진짜 문제:** 운영자(학부생 1인)의 목표는 "실제 외주 수주(사업)"인데,
지금 병목은 *견적 자동화*가 아니라 **신뢰 증명 + 실제 제안(outreach)**이다. 자동 견적
사이트는 그 병목을 직접 풀지 못한다. 운영자 본인의 동기가 이를 드러냈다 —
"뭐라도 만들어두어야 사람들이 외주를 맡길 것 같아서."

## Demand Evidence

- **현재 수요 증거: 없음.** 돈을 내거나 "이거 만들어 달라"고 구체적으로 요청한 사람 0명.
  (운영자 본인이 Q1에서 솔직히 인정 — 이 세션의 가장 중요한 한 줄.)
- 단, **수요에 인접한 강한 신뢰 자산은 이미 보유:**
  - 라이브 배포·실사용 중인 제품 3개 (ClassFileAuto, likelion-sch.com, Hedgehog WebCTF)
  - 멋쟁이사자처럼 14기 풀스택 파트장 (교육 세션 진행) — 비개발 창업팀에 둘러싸인 위치
  - 보안 연구회 인턴 + KISIA·KUCIS 보안 이력 다수
  - 포트폴리오 사이트(kimhands.github.io) + GitHub README 기제작
- **미활용 채널:** 멋사 네트워크 / 해커톤·아이디어톤·창업동아리의 "아이디어는 있는데
  못 만드는 비개발 창업자". 이것이 운영자의 가장 현실적인 첫 3~5 고객 풀.

## Status Quo

- 운영자의 첫 계약 현실 경로(본인 답변): 아는 사람 소개 + 프리랜서 플랫폼 홍보.
- 비개발 클라이언트가 소프트웨어를 외주 줄 때 실제로 사는 것은 "견적의 정확함"이 아니라
  **"이 사람이 끝까지 만들어 줄 것이라는 신뢰"**. 모르는 사람에게 셀프 견적 계산기는
  그 신뢰를 만들지 못한다.

## Target User & Narrowest Wedge

- **타깃 (1차):** 멋사/해커톤/창업동아리에서 운영자를 이미 아는 또는 1-hop 소개되는
  비개발 창업팀 — MVP/웹서비스가 필요하지만 못 만드는 사람들.
- **가장 좁은 웨지:** "학생 창업팀/해커톤 팀을 위한 2주 단위 고정 범위 MVP 개발",
  포트폴리오 가격(첫 2건), 선금 30~50%. 자동 계산기가 아니라 **사람 대 사람 제안**.

## Constraints

- 1인 캐파 제한 → 규모·기한 과도 입력 시 "상담 필요"로 분기(명세서 §4 = 캐파 가드레일 유지).
- **동시 진행 1건 원칙.** 제안 여러 건이 동시 성사돼도 학부생 1인이 2주 MVP를 병렬로
  돌릴 수 없음. 둘째 건은 대기열/일정 뒤로(입력 단계 분기와 별개로 본인 캐파 관리).
- man-day 실제 데이터 없음 → 정밀 견적 엔진을 지금 캘리브레이션 불가.
- 전 과정 바이브코딩(Claude Code) + 운영자 보안 전공.

## Premises (확정)

1. **(수정됨)** 자동 견적 *자체*가 문제가 아니라, **정밀 계산 엔진에 과투자**하는 게
   문제다. 견적은 정밀 엔진이 아니라 **"예산대·범위 대화 시작기"**로 남긴다.
   (Codex 제2 의견이 원전제를 "과장"이라 challenge → 운영자가 수정 채택.)
2. 엔진의 man-day 숫자는 운영자가 아직 갖지 못한 데이터 위에 서 있다. **실제 1~2건을
   해본 뒤에야** §6.3 상수를 역산·보정할 수 있다. 빌드 순서가 거꾸로였다.
3. 이 사이트의 현재 진짜 가치는 "리드 전환 계산기"가 아니라
   **포트폴리오·신뢰 증명물 + 상담 보조 도구**다. (운영자 부분 동의.)

## Cross-Model Perspective (Codex 콜드 리드)

독립적 AI(이 대화 미열람)의 핵심 요약:
- **최강 버전:** 자동견적 사이트가 아니라 "신뢰형 개발 제안 페이지" — 포트폴리오 3개,
  빠른 MVP, 명확한 범위/가격/일정, 쉬운 첫 상담이 핵심. 자동견적은 보조 도구.
- **가장 중요한 신호:** "뭐라도 만들어야 신뢰받는다"는 *오판*. 문제는 견적 자동화가
  아니라 신뢰 부족이고, 필요한 건 추가 제품이 아니라 기존 증거를 팔리게 포장 + 실제 제안.
- **challenge한 전제:** 전제 1은 과장 — 따뜻한 네트워크에서도 견적은 "예산대 대화
  시작기"로 유용. (→ 전제 1 수정에 반영)
- **48시간안:** 랜딩 1페이지(포트폴리오 + 가격 패키지 3개 + 포함/제외 + 2주 가능/불가
  + 상담 CTA), 자동견적은 "예상 예산대"만. Recharts·Supabase·AI·multiplier 5종·
  회원가입은 컷. **빌드보다 30명 아웃리치·10콜·유료 제안 3·선금 1이 목표.**

## Approaches Considered

### Approach A: 신뢰 제안 페이지 우선 (최소 실행)
단일 랜딩 — 포트폴리오 3개 최상단 + 가격 패키지 3개 + 포함/제외표 + "2주 가능/불가"
+ 상담 CTA. 자동견적은 정밀 엔진 제거하고 **"예상 예산대"를 정적 룩업 테이블**로 구현
(선택 → 미리 정해둔 구간 매핑, multiplier 계산 없음). 정밀화는 C단계로 미룸.
공수 S / 리스크 낮음. 명세서 §3(포지셔닝 카피)·§7(집 짓기 UX·포함/제외) 재사용.
못 가진 숫자 의존 안 함.

### Approach B: 명세서 6단계 풀빌드 (원안)
§10 Phase 1~6 전부 — 풀 man-day 엔진 + Recharts + Supabase + AI 정리기 + PDF.
공수 XL / 리스크 높음. 검증 안 된 숫자 위 엔진, 잘못된 채널 최적화, 매출 0 기간 김.

### Approach C: 역순 하이브리드 (창의적)
먼저 실제 1~2건 수주(기존 포트폴리오 + A의 가격 페이지 + 수동 견적) → 실제 단가로
§6.3 상수 보정 → 그 다음 §6 풀 엔진을 진짜 데이터 기반으로 완성(포트폴리오·스케일링
도구). 엔진은 검증의 *보상*. 공수 A(S)→엔진(M) / 리스크 낮음.

## Recommended Approach

**A를 지금 빌드하고 C의 순서로 진화** (운영자는 이 방향을 다운스트림에서 확정 예정).
- 지금: Approach A 페이지 + 즉시 아웃리치.
- 첫 수주 후: 실제 man-day로 §6 엔진 보정·구현(Approach C). "내가 만든 견적 사이트로
  내 외주를 받는다"는 스토리가 그 자체로 포트폴리오가 된다.
- B 단독은 함정(미검증 숫자 + 잘못된 채널 + 긴 무매출 기간).

> 운영자 결정: 본 문서를 `/plan-eng-review`·`/grill-me`로 다듬어 최종 개발 문서를
> 새로 만든 뒤 개발 착수. 최종 A/B/C 확정은 eng-review 단계에서.

## NOT in scope (의도적 연기)
- **정밀 man-day 견적 엔진**(명세서 §6 multiplier·버퍼) — 실제 수주 1~2건으로 상수 보정 후(C단계). 지금은 티어 분류 룩업.

## What I noticed about how you think

- "뭐라도 만들어두어야 사람들이 외주를 맡길 것 같아서"라고 했는데, 정작 이미 라이브
  제품 3개를 배포했고 멋사 풀스택 파트장입니다. 당신은 **만들 줄 아는 사람**인데
  스스로를 "무경험"으로 깎고 있어요. 진짜 격차는 실력이 아니라 *제안*입니다.
- Q1에서 "아직 증거 없음"이라고 솔직히 답한 것 — 대부분은 "관심 많아요"로 포장합니다.
  그 솔직함이 이 세션을 빠르게 핵심으로 데려갔습니다.
- 전제 1을 무조건 받아들이지 않고 "부분적 동의"로 자동견적을 지킨 것, 그리고 Codex의
  challenge를 듣고 수정한 것 — *근거에 따라 입장을 바꾸는* 태도. 좋은 신호입니다.
- 명세서의 "집 짓기 비유 + 포함/별도 전면 노출"은 제품 감각(taste)이 분명합니다.
